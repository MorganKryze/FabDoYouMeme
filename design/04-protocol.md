**`GET /api/users/me/export` response shape**:

Returns a single JSON object containing all personal data held for the requesting user. Intended for GDPR Art. 20 (data portability) requests.

```json
{
  "exported_at": "2026-04-02T10:00:00Z",
  "user": {
    "id": "uuid",
    "username": "alice",
    "email": "alice@example.com",
    "role": "player",
    "consent_at": "2026-01-15T09:30:00Z",
    "created_at": "2026-01-15T09:30:00Z"
  },
  "game_history": [
    {
      "room_code": "WXYZ",
      "game_type_slug": "meme-caption",
      "pack_name": "Classic Memes Vol 1",
      "started_at": "2026-03-27T14:00:00Z",
      "finished_at": "2026-03-27T14:45:00Z",
      "score": 42,
      "rank": 2,
      "player_count": 4
    }
  ],
  "submissions": [
    {
      "round_id": "uuid",
      "room_code": "WXYZ",
      "game_type_slug": "meme-caption",
      "payload": { "caption": "When the CI passes on the first try" },
      "created_at": "2026-03-27T14:10:00Z"
    }
  ]
}
```

Notes:

- `submissions` includes only rounds where `user_id` matches — sentinel-replaced submissions (hard-deleted accounts) are excluded automatically.
- Votes cast by the user are **not** included — they contain no personal data beyond `voter_id` which is already in the user object.
- Packs owned by the user are not included — they are content, not personal data.
- Rate limit: subject to `RATE_LIMIT_GLOBAL_RPM`. No additional throttle needed (endpoint is non-sensitive).
- **Completeness**: this export covers all personal data provided or generated by the user. Not included: session tokens (security-sensitive — revoke via logout instead), magic link tokens (expired within 15 min; no value after use), audit log entries where user acted as admin (their actions on _other_ users, not their own data).

**Admin user search**: `GET /api/admin/users?q={term}` performs a case-insensitive substring match against both `username` and `email`:

```sql
WHERE lower(username) LIKE lower('%' || $q || '%')
   OR lower(email)    LIKE lower('%' || $q || '%')
```

Combines with cursor pagination: `?q=alice&limit=50&after=...`. Clearing the search term returns the full paginated list.

**Pack visibility rules for `GET /api/packs`**:

- Admins see all packs (all visibilities, all statuses)
- Authenticated users see: their own packs (any visibility) + all `public` packs with `status = 'active'`
- `status = 'banned'` packs are excluded from all listings except for the owning admin
- `?game_type_id={uuid}` further filters to packs with ≥1 compatible item (see `03-data.md`)

**Admin notification trigger**: when `visibility` is set to `public` on a new pack (`POST`) or an existing pack (`PATCH`), the backend inserts an `admin_notifications` row of type `pack_published`. When a public pack's items are modified, it inserts `pack_modified`. The pack remains live.

## WebSocket

```plain
WS /api/ws/rooms/:code
```

All messages are JSON with a `type` field and an optional `data` object. Authentication happens during the HTTP upgrade: the session cookie is validated against the `sessions` table. Unauthenticated upgrade requests are rejected with `401` before the WebSocket handshake completes.

### Type Namespace

- **Lifecycle** types (`join`, `start`, `game_ended`, …) are shared across all game types.
- **Game-specific** types are prefixed with the game type slug: `meme-caption:submit`, `trivia:answer`, etc.
- The backend rejects any game-type-prefixed message whose prefix does not match `room.game_type.slug` with `unknown_game_type`.

### Client → Server Messages

| Type            | When                                                 | Payload                                                                    |
| --------------- | ---------------------------------------------------- | -------------------------------------------------------------------------- |
| `join`          | Player enters lobby                                  | `{}` — rejected with `game_already_started` if `room.state != 'lobby'`     |
| `reconnect`     | Player re-establishes connection within grace window | `{}` — triggers `room_state` snapshot instead of `player_joined` broadcast |
| `start`         | Host starts the game                                 | `{}` — rejected with `not_host` for non-host                               |
| `next_round`    | Host advances to next round                          | `{}` — only host; no-op if current round not yet ended                     |
| `ping`          | Keepalive heartbeat                                  | `{}` — sent every `WS_PING_INTERVAL` (default 25s) by client               |
| `{slug}:submit` | Player submits their answer                          | Game-type-specific — see game type protocols below                         |
| `{slug}:vote`   | Player casts vote                                    | Game-type-specific — see game type protocols below                         |

### Server → Client Messages

| Type                 | When                                      | Payload                                                                                                                                                     |
| -------------------- | ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pong`               | Response to client `ping`                 | `{}`                                                                                                                                                        |
| `player_joined`      | Player joins the lobby                    | `{ user_id, username }`                                                                                                                                     |
| `player_left`        | Player disconnects (grace window expired) | `{ user_id, username }`                                                                                                                                     |
| `player_kicked`      | Host kicks a player                       | `{ user_id, username, reason: "kicked" }` — sent to entire room; kicked player's connection is closed                                                       |
| `reconnecting`       | Player enters grace window                | `{ user_id, username }` — other players see "reconnecting…"                                                                                                 |
| `game_started`       | Host starts the game                      | `{ round_count, config }`                                                                                                                                   |
| `round_started`      | New round begins                          | `{ round_number, item: { payload, media_url? }, duration_seconds, ends_at: ISO8601 }`                                                                       |
| `submissions_closed` | Submission phase ends, voting begins      | `{ voting_duration_seconds, ends_at: ISO8601 }`                                                                                                             |
| `vote_results`       | Round scores revealed                     | Game-type-specific — see game type protocols below                                                                                                          |
| `game_ended`         | Final leaderboard                         | `{ reason: "completed" \| "host_disconnected" \| "all_players_disconnected" \| "pack_exhausted", leaderboard: [{ user_id, username, total_score, rank }] }` |
| `room_state`         | Full snapshot sent on reconnect           | `{ state, players, current_round?, config }`                                                                                                                |
| `error`              | Any error from server                     | `{ code: "snake_case_code", message: "..." }`                                                                                                               |

### Security & Reliability

**Message size limit**: `SetReadLimit(WS_READ_LIMIT_BYTES)` — default 4 KB per message. Clients exceeding this are disconnected.

**Per-connection rate limit**: `WS_RATE_LIMIT` messages/second (default 20). Clients exceeding this are disconnected.

**Heartbeat**: client sends `{"type":"ping"}` every `WS_PING_INTERVAL` (default 25s). Server replies `{"type":"pong"}`. Server sets a `WS_READ_DEADLINE` (default 60s) read deadline on each connection, reset on every received pong. A connection that does not pong within the deadline is considered dead and cleaned up.

**Reconnection & grace window**:

- On disconnect, player is marked `reconnecting` for `RECONNECT_GRACE_WINDOW` (default 30s).
- Other players receive `{"type":"reconnecting","data":{"user_id":"..."}}` instead of `player_left`.
- If player reconnects within the window: server sends `room_state` snapshot; client sends `reconnect` (not `join`).
- If grace window expires: `player_left` is broadcast and their pending turn is skipped.

**Client reconnection backoff**: the `WsState` class uses exponential backoff with jitter: `min(30, 2^(attempt-1)) + random(0, 1)` seconds, up to 10 attempts. After 10 failed attempts, `status` transitions to `'error'` and the user must manually retry. The UI shows "attempt N / 10" in the reconnecting banner.

**Quorum on all-disconnect**: if all players simultaneously enter the grace window, the round timer is paused. If ≥1 player reconnects within the window, the game resumes. If none reconnect, `game_ended` is broadcast with `reason: "all_players_disconnected"` and the room moves to `finished`.

**Host disconnect**:

- During `lobby`: if host's grace window expires, room moves to `finished`. Players see "host left".
- During `playing`: if host's grace window expires, `game_ended` is broadcast with `reason: "host_disconnected"`. Room moves to `finished`. No host transfer — see ADR-009 in `ref-decisions.md`.

**Host disconnect UX contract**: clients that receive `game_ended` with `reason = "host_disconnected"` MUST display "Host left — game ended" as a prominent banner above the leaderboard (not a generic disconnect screen). The leaderboard is still shown below the banner if scores are available.

### Timer Contract

Every timed WS event includes both fields:

```json
{
  "type": "round_started",
  "data": {
    "round_number": 3,
    "duration_seconds": 60,
    "ends_at": "2026-03-27T14:05:00.000Z",
    "item": { ... }
  }
}
```

- **`duration_seconds`**: the server's enforcement parameter. The hub's timer goroutine uses this to schedule `submissions_closed`. Authoritative.
- **`ends_at`** (ISO 8601, server clock): for client countdown display only. The client calculates remaining time as `ends_at - Date.now()` on each tick to avoid clock drift.
- **Client button disable**: the `Submit`/`Vote` button is disabled when `Date.now() >= Date.parse(ends_at)`. This is a UX convenience — the server will reject late submissions regardless of the client's button state.

### Game Type Protocol: `meme-caption`

Gameplay: host shows an image with a prompt → players write captions → all captions shown anonymously → players vote for the best → author revealed in results.

#### Client → Server

| Type                  | Payload                   | Notes                                                                                   |
| --------------------- | ------------------------- | --------------------------------------------------------------------------------------- |
| `meme-caption:submit` | `{ caption: string }`     | Max 300 chars; trimmed; rejected after `submissions_closed` (`submission_closed` error) |
| `meme-caption:vote`   | `{ submission_id: uuid }` | One vote per player per round; see vote integrity contract below                        |

#### Server → Client

| Type                             | Payload                                                                                                                              | Notes                                                                                          |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| `meme-caption:submissions_shown` | `{ submissions: [{ id, caption }] }`                                                                                                 | Authors **hidden** during voting; own submission included but client visually distinguishes it |
| `meme-caption:vote_results`      | `{ submissions: [{ id, caption, author_username, votes_received, points_awarded }], round_scores: [{ user_id, username, points }] }` | Authors revealed; cumulative scores updated in `game_ended`                                    |

**Scoring model**: one point per vote received. Tied submissions share points equally (both receive full points — no tiebreaker). Scoring logic lives in `handler.CalculateRoundScores`.

**Submission window**: accepted from `round_started` until `submissions_closed`. Closed after `round_duration_seconds` OR when all players have submitted — whichever first.

**Voting window**: open after `submissions_shown` until `voting_duration_seconds` elapse OR all players have voted — whichever first.

**Author anonymity**: `BuildSubmissionsShownPayload` omits `author_id` and `author_username`. `BuildVoteResultsPayload` includes both.

## Pagination

All list endpoints that can return unbounded rows support cursor-based pagination (avoids "page drift" on concurrent inserts).

| Parameter | Type   | Description                                          |
| --------- | ------ | ---------------------------------------------------- |
| `limit`   | int    | Max rows to return (default 50, max 100)             |
| `after`   | string | Opaque cursor from previous response's `next_cursor` |

**Response envelope**:

```json
{
  "data": [ ... ],
  "next_cursor": "opaque_base64_string_or_null",
  "total": 142
}
```

`next_cursor` is `null` when no more pages. `total` is the full count before pagination.

The cursor encodes `{ id, created_at }` of the last row, base64-encoded. Clients treat it as opaque.

**Paginated endpoints**: `GET /api/admin/users`, `GET /api/admin/invites`, `GET /api/packs`, `GET /api/packs/:id/items` (ordered by `position ASC`), `GET /api/packs/:id/items/:item_id/versions` (ordered by `version_number DESC`), `GET /api/admin/notifications` (ordered by `created_at DESC`).

## Item Reorder

`PATCH /api/packs/:id/items/reorder` accepts a full position map:

```json
{
  "positions": [
    { "id": "uuid-a", "position": 1 },
    { "id": "uuid-b", "position": 2 }
  ]
}
```

Uses the two-pass strategy (ADR-010): shift all to `position + 10000`, then set final values in a single transaction.

**Validation**: request must include all items in the pack (no partial reorder), positions must be unique, start at 1, and contain no item IDs from a different pack. Returns `422 positions_invalid` on any violation (see `ref-error-codes.md`).

## User Hard-Delete Protocol

`DELETE /api/admin/users/:id` hard-deletes the user. The following steps run in a **single transaction** in this order:

1. `SELECT username, email FROM users WHERE id = $target` — capture PII in local variables
2. `INSERT INTO audit_logs (admin_id, action, resource, changes) VALUES ($admin, 'hard_delete_user', 'user:$target', '{"username":"...", "email":"..."}'::jsonb)` — PII recorded BEFORE deletion so the audit trail is meaningful
3. `UPDATE submissions SET user_id = $sentinel WHERE user_id = $target` — sentinel UUID is `00000000-0000-0000-0000-000000000001` (ADR-006)
4. `UPDATE votes SET voter_id = $sentinel WHERE voter_id = $target` — preserve game history; unlink voter identity
5. `DELETE FROM users WHERE id = $target` — cascades: `sessions`, `magic_link_tokens`, `room_players` (ON DELETE CASCADE); sets NULL on: `invites.created_by`, `audit_logs.admin_id`, `game_packs.owner_id`, `admin_notifications.actor_id`

If any step fails, the transaction rolls back. The audit log entry is only committed if the deletion succeeds.

**Recommended approach**: deactivate first (`is_active = false`) and reserve hard delete for explicit GDPR erasure requests. Document this distinction clearly in the admin UI: "Deactivate prevents login. Delete permanently removes all personal data."

## Game Engine & Extensibility

### Overview

Game types are self-contained handler units. Adding a new game type requires:

1. A new migration seeding `game_types` with the slug, name, and config
2. A new backend handler package under `internal/game/types/{slug}/`
3. A new frontend plugin folder under `src/lib/games/{slug}/`
4. New frontend route(s) if the game type needs unique views

**No schema or WebSocket protocol changes are needed.** Game-specific content lives in JSONB `payload` fields and prefixed WS message types.

### GameTypeHandler Interface

```go
// internal/game/handler.go

type Round struct {
    ID                 uuid.UUID
    RoomID             uuid.UUID
    ItemID             uuid.UUID
    RoundNumber        int
    StartedAt          *time.Time
    EndedAt            *time.Time
    ItemPayload        json.RawMessage
    ItemPayloadVersion int
}

type Submission struct {
    ID      uuid.UUID
    UserID  uuid.UUID
    Payload json.RawMessage
}

type Vote struct {
    SubmissionID uuid.UUID
    VoterID      uuid.UUID
    Value        json.RawMessage
}

type GameTypeHandler interface {
    // Slug returns the game type slug matching game_types.slug (e.g. "meme-caption").
    Slug() string

    // SupportedPayloadVersions returns which item payload versions this handler can process.
    // Items with an unsupported version are skipped (next available item used instead).
    // Exposed via GET /api/game-types/:slug and used for pack compatibility filtering.
    SupportedPayloadVersions() []int

    // SupportsSolo returns whether this game type supports solo mode.
    // Enforced at POST /api/rooms (solo_mode_not_supported error if false).
    SupportsSolo() bool

    // ValidateSubmission checks that the submission payload is valid for this game type and round.
    // Returns a descriptive error sent to the client as {"code":"invalid_submission",...}.
    ValidateSubmission(round Round, payload json.RawMessage) error

    // ValidateVote checks that the vote payload is valid.
    // The hub has already verified: (a) voting phase is open, (b) no duplicate vote exists.
    // This method must additionally verify: voterID != submission.UserID (self-vote prevention),
    // and any game-type-specific payload validity.
    // Returns error sent as {"code":"invalid_vote",...} or {"code":"cannot_vote_for_self",...}.
    ValidateVote(round Round, submission Submission, voterID uuid.UUID, payload json.RawMessage) error

    // CalculateRoundScores aggregates votes into per-user point awards.
    // Returns a map of userID → points earned in this round.
    // Called after voting closes; results are broadcast in vote_results.
    CalculateRoundScores(submissions []Submission, votes []Vote) map[uuid.UUID]int

    // BuildSubmissionsShownPayload returns the data for the {slug}:submissions_shown event.
    // Implementations control what is revealed during voting (e.g. captions shown, authors hidden).
    BuildSubmissionsShownPayload(submissions []Submission) (json.RawMessage, error)

    // BuildVoteResultsPayload returns the data for the {slug}:vote_results event.
    // Implementations control what is revealed after voting (e.g. authors revealed, scores shown).
    BuildVoteResultsPayload(submissions []Submission, votes []Vote, scores map[uuid.UUID]int) (json.RawMessage, error)
}
```

### Vote Integrity Contract

The hub enforces vote integrity **before** calling `handler.ValidateVote`:

1. Check `room.state == 'playing'` and `round.phase == 'voting'` — if not, return `voting_closed`
2. Query `votes` table: `WHERE submission_id IN (SELECT id FROM submissions WHERE round_id = $round_id) AND voter_id = $user_id` — if row found, return `duplicate_vote`
3. Delegate to `handler.ValidateVote` — checks self-vote and game-type-specific payload validity

Only if all three checks pass: insert the vote row. This ensures vote integrity is enforced consistently regardless of which game type handler is active.

### Payload Version Handling

When loading items for a round:

1. Check `item.payload_version`
2. If not in `handler.SupportedPayloadVersions()`, log an error and skip — select the next available item in the pack instead
3. Use version-appropriate parsing logic inside the handler

**Edge case: zero supported items remaining mid-game**. Before starting each round, the hub verifies that at least one unplayed item in the pack has a compatible `payload_version`. If none remain, broadcast `game_ended` with `reason: "pack_exhausted"` and move the room to `finished`. This should only occur if item versions were changed after room creation (which the API prevents), but is handled defensively.

When a new payload version is released:

- Deploy updated handler declaring `SupportedPayloadVersions: []int{1, 2}` (both old and new)
- New items use version 2; existing version-1 items continue to work
- No DB migrations needed — old data is immutable

### Game Type Lifecycle & Versioning

Game types are **immutable once seeded** in production. If mechanics change significantly, seed a new slug (e.g., `meme-caption-v2`) rather than mutating the existing entry. The `game_types.version` column is informational only and does not affect dispatch logic.

### In-Memory State and Restart Behaviour

The WebSocket hub holds all active room state in **memory only**. Persistent state (room, round, submissions, votes) is written to PostgreSQL as events occur, but hub runtime state is not recoverable.

**On restart**: all active WebSocket connections drop and in-progress games are lost. The backend moves orphaned `playing` rooms to `finished` on startup (see `06-operations.md`). The frontend must handle this: if a `reconnect` is rejected because the hub no longer exists, show:

> "The server restarted and this game could not be recovered. The room has been marked as finished."

This is a deliberate simplicity trade-off — full game-state persistence adds significant complexity for minimal benefit at self-hosted scale with planned maintenance windows.
